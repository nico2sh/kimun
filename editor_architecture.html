Kimun TUI — Text Editor Architecture

Render and input pipeline of TextEditorComponent. Covers the two backends (in-process ratatui-textarea and external neovim over RPC), the layered view caches keyed on text_revision, the autocomplete + autosave async pipelines, and the rules that protect against concurrent writes / stale completions.

2. Frame render pipeline

Every frame, EditorScreen::render calls TextEditorComponent::render. The component picks lines + cursor from the active backend, drives MarkdownEditorView::update (which gates its caches on the buffer revision counter), then calls view.render and finally overlays the autocomplete popup.

Gate 1 incremental path. When text_revision advances, Gate 1 first attempts try_incremental_parse: it runs compute_damage_range(lines_snapshot, lines, cursor.0) to find the row range that differs (cursor-row fast path for single-char edits; LCP/LCS scan for everything else), then applies a hybrid widener.

Widener (three-tier). The widener tries three boundary sources in order:

  1. Strict. expand_to_reset_boundary(parsed_buffer.reset_boundaries, …): reset_boundaries are rows where pulldown-cmark's parser state is provably reset (lazy_depth[r] == 0 && kinds[r] == Blank), so a slice between two reset boundaries is guaranteed equivalent to a fresh parse over that range — no post-slice verification needed. Carries ~98% of incremental successes in dogfooding (fast_path_share = 98.5%).
  2. Intra-construct. expand_to_reset_boundary(parsed_buffer.intra_construct_boundaries, …): rows where pulldown emits End(Item) / End(BlockQuote) such that splicing across the row preserves per-row rendered output (kinds, elements.len(), content_vis) even though the surrounding lazy construct is still open (lazy_depth[b] > 0). Unlocks incremental typing inside loose lists (heavy_lists_loose.md: 493 µs → ~36 µs, ~13× speedup). Sparse heuristic (MIN_INTRA_BOUNDARY_GAP = 2) keeps the set O(items / 2).
  3. Heuristic. widen_to_safe(parsed_buffer.kinds, damaged): stops at Blank or Plain rows. Last-resort fallback when both reset-boundary widener attempts return FullRebuild. Carries 0.6% of successes — kept as a safety net for shapes the boundary sets haven't characterised.

Lazy-depth-aware structural guards. Before either widener runs, try_incremental_parse bails to a full parse when the damaged row trips any of:

  • The damaged row was a structural marker pre-edit (FenceMarker, SetextUnderline, IndentedCode, HtmlBlock) whose extent can shift beyond the widening window.
  • The new content introduces or removes a structural-marker prefix (looks_like_fence_marker, looks_like_setext_underline, looks_like_indented_code, looks_like_html_block_opener).
  • parsed_buffer.lazy_depth[row−1..=row+1] contains any positive entry — an open lazy-continuable construct (List, BlockQuote, IndentedCode, HtmlBlock) at or next to the damaged row can lazy-extend across the splice in ways the slice cannot capture. Conditionally relaxed when BOTH: (a) damaged row's old kind is ListMarker, AND (b) lazy_depth[row] == 1 (top-level list, not nested inside a blockquote / outer list). Marker-row content edits cannot change row+1's classification (the post-pass ListMarker→ListContinuation propagation is deterministic from the prev row's kind, which the splice updates in lockstep). Marker/blank flips are caught by the surrounding guards (looks_like_* runs before this check; blank-transition runs after), so the relax only ever fires on pure content edits. Initial relaxation also accepted ListContinuation + nested depths; both unlocks reverted after the 100k proptest soak exposed downstream-row-classification flips. Relaxation rate tracked in widener_metrics::lazy_guard_relaxed.
  • The damaged row flips between blank and non-blank with a non-blank neighbour — the pre-edit reset boundary at that row becomes invalid post-edit because a paragraph (or other lazy construct) can re-extend across the row.

On success, ParsedBuffer::parse_range(lines, widened) parses the slice as a standalone document and splices it into parsed_buffer; the splice also updates lazy_depth, reset_boundaries, and intra_construct_boundaries.

Post-slice verify policy. A field-by-field equality check (slice's kinds/elements.len/content_vis vs parent's pre-edit values, skipping damaged rows) runs only on non-Strict paths and only under cfg(debug_assertions) + KIMUN_VIEW_VERIFY_INCREMENTAL=1. The Strict path is provably equivalent to a fresh parse so it skips verify entirely. Both fallback paths were demoted from release after their respective proof-out windows (v2 phase-7 for Heuristic: 1581 attempts, 0 hits; intra-construct PR 3: 1011 attempts + 600k proptest cases, 0 hits). The env-gated debug-only verify stays as the proptest harness's correctness backstop. If a release-build divergence ever surfaces, re-enable in release while a guard is added.

See openspec changes parse-reset-boundaries, parse-reset-boundaries-v2, and intra-construct-reset-boundaries.

3. Backend abstraction

Textarea backend sync

  • ratatui_textarea::TextArea owns the buffer in-process.
  • Key events go through handle_textarea_key which routes through a shortcut match (cursor moves, Ctrl+Z/Y, Ctrl+X cut, etc.) and the catch-all input_without_shortcuts.
  • text_revision is bumped only on confirmed text mutations (return value of undo(), cut(), input_without_shortcuts(), etc.).
  • Autocomplete is enabled. The host snapshot reads ta.lines() and cursor_tuple at the moment of handle_input.
  • is_dirty() = saved_text_revision != Some(self.text_revision).

Nvim backend RPC

  • NvimBackend owns a child process and a long-running tokio refresh task that polls Lua state on flush events.
  • The refresh task writes into a Mutex<NvimSnapshot>: lines, cursor, dirty flag, content_gen, mode, selection.
  • Keys are forwarded via nvim.handle_key (RPC); replies update the snapshot asynchronously.
  • text_revision is bumped conservatively on every Nvim key — there's no synchronous way to know whether a key mutated the buffer, and the autosave correctness check depends on a monotonic counter.
  • Autocomplete is not wired up on Nvim.
  • is_dirty() reads snap.dirty directly.

4. Revision counters

Two counters drive every cache + correctness decision. Both are u64 with monotonic-non-zero semantics (start at 1, skip 0 on wraparound):

CounterBumped byConsumed by
text_revision bump_text — only when buffer truly changes (Textarea path returns bool from cut / undo / input_without_shortcuts / …). On Nvim: conservatively bumped per key.
  • is_dirty (Textarea path)
  • view.update as the cache-invalidation key
  • AutocompleteHost::text_revision → exclusion-zone cache key
  • spawn_autosave → AutosaveCompleted.saved_revision
  • handle_input uses delta to choose sync vs refresh_if_open
edit_generation Both bump_cursor and bump_text — cursor moves bump it; pure no-ops don't. View-cache-adjacent invariants where cursor moves matter (last_layout_cursor tracking).

Sentinel: text_revision == 0 — reserved as the AutocompleteHost::text_revision "do not cache" signal. SearchBoxHostSnapshot returns 0 permanently. The controller treats 0 as an unconditional cache miss and skips persisting the rebuilt zones so sentinel hosts don't churn the slot.

5. View cache layers

All caches live on MarkdownEditorView. Two gates decide what to rebuild on each update():

Cache Type Rebuilt when Cost when rebuilt
lines_snapshot Vec<String> Gate 1 (text revision changed OR line count drift) O(N) clone
fence_ranges Vec<Range<usize>> Gate 1 O(N) derived from parsed_buffer.kinds via fence_ranges_from_kinds — single pass, no duplicate string scan
parsed_buffer ParsedBuffer { lines: Vec<ParsedLine>, kinds: Vec<LineConstructKind> } Gate 1 (full or incremental splice) O(N) full parse or O(widened_range) incremental; row-count invariant enforced by ParsedBuffer::splice
rendered_cache Vec<Vec<bool>> Gate 2 content branch (full) or partial (two cursor rows) O(N) chars (full) / O(2 × line_chars) (partial)
layout WordWrapLayout Gate 2 (content / width / cursor row / element expansion) O(N) lines × O(chars/line)
last_cursor_screen Option<(u16, u16)> Every render (cheap) O(1) — anchor for autocomplete popup

View cache layers. MarkdownEditorView holds a parsed_buffer: ParsedBuffer — a single struct with two equal-length vectors: lines: Vec<ParsedLine> (one per logical row, carrying per-character visibility masks, element ranges, and image placeholders) and kinds: Vec<LineConstructKind> (the coarse classification — Blank, Plain, FenceMarker, FenceContent, ListMarker, etc. — that drives safe-boundary widening). ParsedBuffer::splice(range, other) enforces the row-count invariant: after every splice, lines.len() == kinds.len() == row_count_of_buffer. The view also caches fence_ranges (derived from parsed_buffer.kinds via fence_ranges_from_kinds — single fence detector, no duplicate string scan) and lines_snapshot (the previous-frame buffer used by the damage-diff in Gate 1).

8. Cross-cutting invariants

  • Cursor moves never dirty the buffer. Pinned by cursor_move_does_not_dirty_buffer.
  • No-op shortcuts (empty-stack undo/redo, word-boundary delete) bump neither counter. Pinned by empty_stack_undo_redo_does_not_dirty_or_bump_revision.
  • Initial text_revision is non-zero. Avoids colliding with the autocomplete cache sentinel. Pinned by fresh_editor_text_revision_is_nonzero.
  • Stale Nvim cursors cannot panic the render thread. render() uses .get() on parsed_cache and visual_lines. Pinned by render_does_not_panic_on_stale_cursor_past_line_count.
  • Sentinel hosts (revision 0) neither read nor write cached_text. Pinned by sentinel_revision_does_not_populate_or_consult_cache.
  • await_or_abort aborts a hung task on timeout. Dropping a tokio JoinHandle only detaches — it does NOT cancel. Without the abort, a second vault.save_note could race the stuck one on the same path. Pinned by await_or_abort_aborts_long_task_on_timeout + await_or_abort_returns_true_when_task_completes.
  • mark_saved vs mark_saved_at_revision asymmetry is intentional. Sync mark_saved clobbers on divergence (caller is authoritative). Async mark_saved_at_revision no-ops on divergence (stale completion must not undo a newer clean state set by a racing sync path).
  • Main loop drain breaks on screen_generation change. Queued events from the old screen instance can't leak into a new screen — even if both have the same ScreenKind (Editor→Editor on follow-link).
